# Accessibility Statement

## Purpose
This document defines the current platform-level accessibility position for Monyawn.

It is intended to support product design, user trust, and buyer review. Public claims, VPAT statements, or contract representations should be aligned with actual product testing and formal review before publication.

## Accessibility Intent
Monyawn is designed to be an accessibility-first guided workflow product.

The platform should prioritize:
- keyboard usability
- screen-reader compatibility
- readable structure and content
- strong contrast and focus visibility
- mobile usability
- support for zoom and text scaling

## Product Direction
Monyawn aims to support a guided workflow experience that is understandable for zero-knowledge users and usable across a broad range of abilities and assistive technologies.

## Minimum Accessibility Expectations
The product should support:
- keyboard-only completion of primary workflows
- visible focus indicators
- clear labels and landmarks
- readable content structure
- support for zoomed interfaces
- responsive layouts without avoidable horizontal scrolling

## Testing Expectations
Accessibility testing should include:
- keyboard testing
- screen-reader review
- responsive and zoom review
- usability checks on high-priority flows

## Accessibility Support
Monyawn should provide a documented path for:
- reporting accessibility issues
- requesting support or accommodation guidance
- prioritizing remediation of severe accessibility barriers

## Governance
Primary owners:
- Accessibility Lead
- UX Lead

Consulted owners:
- QA / Reliability Lead
- Frontend Engineer
- Product Lead

## Summary
Accessibility is not a post-launch polish task for Monyawn. It is a core operating requirement. If a major workflow is not usable accessibly, it should not be treated as complete.
