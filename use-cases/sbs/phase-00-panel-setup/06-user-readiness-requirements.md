# User Readiness Requirements

## Purpose
This document defines what end users need from Monyawn and what Monyawn needs from end users so the product can be used safely, clearly, and successfully.

It is intended to support onboarding, trust, accessibility, user responsibilities, and product expectations for both individual and organization-sponsored users.

## What Users Need From Monyawn

| Area | What users need | Why it matters | What Monyawn should provide |
|---|---|---|---|
| Clarity | Plain-language instructions | Zero-knowledge users should not need expert context | Guided step-by-step onboarding |
| Accessibility | Keyboard, screen reader, zoom, mobile support | Users may rely on assistive tech or limited devices | Accessibility-first interface and support |
| Trust | Clear explanation of how AI is used | Users need to know what is generated versus sourced | Visible evidence, rationale, and confidence |
| Privacy | Clear data handling terms | Users upload sensitive career materials | Privacy notice and deletion rights |
| Data custody | Clear explanation of what is saved locally versus exported by the user | Users need to understand loss risk and ownership of their own package files | Local-only storage warning and export guidance |
| Control | Ability to edit and approve outputs | Users remain responsible for real-world actions | Review and approval gates |
| Continuity | Save and resume | Users may need multiple sessions or devices | Resumable progress |
| Support | Help when blocked | Guided products still need recovery paths | Help center and support channels |
| Safety | Boundaries on risky outputs | Users should not be encouraged to misrepresent themselves | Truthfulness and review rules |

## What Monyawn Needs From Users

| Area | What Monyawn needs from users | Why it matters |
|---|---|---|
| Accurate inputs | Honest resumes, job descriptions, notes, and corrections | Output quality depends on source quality |
| Review participation | Approval of key outputs before use | Human oversight is part of the product model |
| Account security | Reasonable password and account practices, or SSO use where provided | Protects user data |
| Lawful use | Compliance with applicable law and employer obligations | Prevents misuse |
| Respect for truthfulness | No intentional use to fabricate skills or credentials | Protects user and platform integrity |
| Proper data handling | Avoid uploading data the user has no right to share | Protects privacy and contractual obligations |
| Export discipline | Save and export their work if they want durable retention | Local-only systems can lose data if the user does not export it |

## User Eligibility
Monyawn should clearly state:
- who may create an account
- minimum age or legal capacity requirements
- whether organization-sponsored accounts are managed by an employer, school, or program
- whether some features are limited by geography, contract type, or subscription plan

## Supported User Types
- individual self-serve users
- organization-sponsored users
- internal workforce program users
- managed-service users supported through AGaaS

## User Responsibilities
Users should agree that they will:
- provide truthful information to the best of their knowledge
- review outputs before using them externally
- avoid using the system to misrepresent experience or identity
- protect login credentials
- report suspected account compromise
- respect employer, customer, or third-party confidentiality obligations

## Monyawn Responsibilities To Users
Monyawn should commit to:
- providing clear step-by-step guidance
- distinguishing sourced information from generated output
- offering save and resume behavior
- providing accessible interaction patterns
- making support and reporting channels available
- documenting major data-handling and AI-usage practices
- warning users clearly when local data may be lost and when export is recommended

## Data And Privacy Requirements For Users
Users should be told:
- what data they upload
- what data is generated by the system
- who can access their data
- how long data is retained
- how data can be exported or deleted
- whether data is used for model training
- how organization-sponsored access may affect visibility
- whether data is stored only locally on their device unless they export it

## AI Transparency Requirements For Users
Users should be told:
- where AI is used
- that outputs can be wrong or incomplete
- that evidence and confidence are shown where applicable
- that high-stakes outputs require review
- that the system is assistive, not a substitute for legal, tax, or employment counsel
- that coaching, candidate story, and compensation education are supportive layers and may be skipped

## Accessibility Commitments To Users
Monyawn should publish:
- supported browsers and devices
- keyboard support expectations
- screen reader support expectations
- accessibility contact method
- accommodation or issue-reporting method

## Minimum User-Facing Policies
Monyawn should maintain user-facing versions of:
- terms of service
- privacy policy
- acceptable use policy
- accessibility statement
- AI usage and limitations notice
- support policy

## User Onboarding Checklist

| Step | What the user sees | What the user must do |
|---|---|---|
| 1. Account start | Product purpose and trust summary | Confirm they want to begin |
| 2. Policy review | Terms, privacy, accessibility, AI usage summary | Accept required terms |
| 3. Profile setup | Basic identity and workflow context | Provide core profile details |
| 4. Source upload | Resume, job description, and notes | Upload or add materials |
| 5. Data confirmation | Extracted details and inferred structure | Correct mistakes |
| 6. Guided workflow | Wizard steps and progress | Complete steps at their own pace |
| 7. Output review | Drafts, recommendations, rationale | Review, edit, and approve |

## User Risk Areas

| Risk | Why it matters | Monyawn response |
|---|---|---|
| User trusts AI too much | High-stakes outputs may be wrong or incomplete | Show review points and confidence |
| User uploads restricted data | Could create privacy or contractual issues | Publish upload guidance and warnings |
| User skips review | Could send unapproved or inaccurate materials | Require review for critical outputs |
| User experiences accessibility barriers | Product adoption and equity suffer | Provide tested accessible flows and support |
| User loses context mid-flow | Career workflows often span multiple sessions | Provide save and resume |
| User loses local data | Local storage can be cleared or lost | Warn early and encourage export handoff packages |

## Organization-Sponsored User Requirements
If a user is part of an employer, school, government program, or workforce organization, Monyawn should clarify:
- whether the sponsoring organization can view account activity
- whether uploaded artifacts are visible to admins
- whether the sponsor controls retention or deletion
- what support path is available for sponsored users

## Summary
Users should be able to answer these questions before they rely on Monyawn:
- What is this product helping me do?
- What do I need to provide?
- What does the AI do versus what do I still control?
- What happens to my data?
- What do I need to do if I want to keep my work safely?
- What accessibility support exists if I need it?
- What am I required to review before I act?

If those answers are not obvious, onboarding is not ready.
