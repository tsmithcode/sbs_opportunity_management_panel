# Buyer Diligence Workflow

## Purpose
This document defines the operating workflow for handling premium enterprise buyer diligence from first request through response, escalation, and closure.

It is intended to help Monyawn respond consistently, preserve current product truth, and prevent unsupported commitments.

## Workflow Principles
- start from current source-of-truth documents
- route quickly when a request crosses into controlled domains
- keep buyer-facing language calm, exact, and non-fabricated
- distinguish clearly between what can be answered now and what requires review

## Workflow Stages

| Stage | Goal | Primary Owner | Output |
|---|---|---|---|
| 1. Intake | Understand what the buyer is asking for | Customer Assurance / Security Questionnaire Lead | Diligence request log and categorized question set |
| 2. Triage | Separate answerable questions from escalation-required questions | Customer Assurance / Security Questionnaire Lead | Routed diligence queue |
| 3. Source review | Answer directly from current approved docs where possible | Trust Center / Public Documentation Owner | Draft response set backed by source materials |
| 4. Controlled review | Escalate security, privacy, sovereignty, legal, pricing, support, or deployment issues | Controlled-domain owner by topic | Reviewed answers or decision to defer/decline |
| 5. Consolidation | Bring all approved answers into one buyer-ready response package | Customer Assurance / Security Questionnaire Lead | Consolidated diligence response |
| 6. Final check | Ensure no answer overstates current posture | Repository Steward / Documentation Systems Lead | Final consistency check |
| 7. Delivery | Send approved packet or answer set to buyer | CRO / Revenue Lead or approved account owner | Buyer-facing diligence response |
| 8. Follow-up | Capture missing recurring questions and document gaps | Benchmarking And Evaluation Lead | Improvement list and trust-center backlog |

## Triage Rules

### Answer directly when:
- the answer exists in current source-of-truth docs
- the answer does not imply unsupported hosting, compliance, pricing, or legal commitments
- the answer does not require product, legal, or controlled-domain interpretation beyond what is documented

### Escalate when:
- the question touches security beyond published docs
- the question requests residency, sovereignty, or deployment commitments
- the question asks for contract or pricing changes
- the question asks for compliance or certification claims not already documented
- the question would change support, retention, or data-use posture

## Diligence Completion Criteria
- buyer received a response set grounded in current documents
- escalated items were answered, deferred, narrowed, or declined by the correct owners
- no buyer-facing answer overstated the current product posture
- recurring new questions were logged for future document improvement

## Common Failure Modes To Avoid
- answering from memory instead of source docs
- letting commercial urgency override controlled-domain review
- implying hosted, sovereign, or compliant posture not currently documented
- mixing exploratory future direction with present product commitments

## Summary
The premium diligence workflow is simple on purpose: intake, triage, answer from truth, escalate where needed, consolidate, check, deliver, and learn. The discipline is what makes the buyer experience feel premium.
